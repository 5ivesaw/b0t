No final-runtime scripted combo.

10. TRAINING ACCELERATION STRATEGY
══════════════════════════════════════════════════════════════════════════════

Pure random reinforcement learning in full real-time Bedwars is unacceptable.

Pure human imitation is also insufficient.

Use a mixture of:

1. Automated teacher generation
2. Structured human telemetry
3. Supervised skill learning
4. Scenario randomization
5. Snapshot branching
6. Validated mechanics simulation
7. Reinforcement-learning fine-tuning
8. Population/self-play evaluation
9. Real-client correction

──────────────────────────────────────────────────────────────────────────────
10.1 AUTOMATED TEACHERS
──────────────────────────────────────────────────────────────────────────────

Create training-only teachers.

Aim teacher:

- Receives exact positions, hitboxes, velocity, latency estimate.
- Produces desired mouse-equivalent yaw/pitch deltas.
- Generates examples across randomized orientation and movement.

Navigation teacher:

- Receives terrain and destination.
- Produces immediate movement labels.
- May use A* or another planner during training only.
- Generates paths with gaps, obstacles, height changes, and interruptions.

Placement teacher:

- Finds valid support blocks.
- Selects candidate face.
- Computes hit point.
- Checks reach.
- Checks collision.
- Produces legitimate rotation and use timing.

Bridging teacher:

- Uses placement geometry.
- Generates straight, diagonal, interrupted, and recovery cases.
- Randomizes orientation, velocity, missing blocks, inventory, and target distance.

Combat teachers:

Use multiple styles rather than one narrow controller:

- Direct tracker
- Smooth tracker
- Aggressive strafing
- Defensive spacing
- Edge-safe fighter
- Predictive interceptor

Inventory/shop teacher:

- Produces labels from resource state, objective, and available offers.
- Verifies transaction result.

Tactical search teacher:

- Evaluates several candidate short-horizon objectives.
- Scores bed risk, death risk, resource value, travel cost, opponent strength,
  and objective probability.
- Produces tactical labels for distillation.

The final runtime policy must not call these teachers.

──────────────────────────────────────────────────────────────────────────────
10.2 HUMAN STRUCTURED TELEMETRY
──────────────────────────────────────────────────────────────────────────────

When the user plays normally, capture:

- Observation snapshot
- Exact key states
- Mouse deltas
- Selected slot
- Attack/use actions
- Current GUI state
- Events after the action
- Outcome
- Timestamp
- Episode metadata

No screen frames.

Correct for human reaction delay.

Do not blindly pair the action at time t with only the state at time t.

Evaluate temporal offsets.

Include recent observation history.

Prevent action imbalance.

Do not allow “hold W” to dominate training simply because it appears most often.

Use:

- Event-centred sampling
- Rare-action weighting
- Balanced batches
- Separate action losses
- Sequence modelling
- Failure/recovery oversampling

Automatically segment telemetry into:

- Navigation clips
- Bridging clips
- Combat clips
- Shopping clips
- Resource clips
- Bed attack clips
- Bed defence clips
- Recovery clips
- Tactical decisions

Human telemetry supplements automated teachers.

It is not the only source.

- Bed defence strength

Use one real state to generate many focused training cases.

Do not claim simulator equivalence until validated.

10.5 DIFFERENTIAL VALIDATION
──────────────────────────────────────────────────────────────────────────────

The simulator is not trusted automatically.

- Grounded state
- Jump trajectory
- Landing tick

- Fall event
- Block placement
- Reach
- Hit timing
- Damage
- Knockback

Create tolerances.

Save every mismatch as a regression case.

Do not expand simulator usage until the relevant mechanic passes differential tests.

10.6 REINFORCEMENT LEARNING
──────────────────────────────────────────────────────────────────────────────

Use reinforcement learning after basic mechanics have been learned from teachers
or structured telemetry.

Use it for:

- Fine-tuning
- Recovery
- Strategy
- Combining skills
- Adaptation to varied opponents
- Improving beyond teacher limitations

Do not ask random exploration to rediscover all Minecraft mechanics.

Choose a standard, inspectable algorithm.

Document:

- Observation space
- Action space
- Reward terms
- Reward scale
- Episode termination
- Curriculum
- Evaluation seeds
- Training/evaluation separation

Prevent reward exploits.

Render representative behaviour.

Inspect failures rather than relying only on reward curves.
